## 2026-07-23 (vol.006 訂正・発行後のCIログ調査より)

上の vol.006 の項に書いた cafe 候補ゼロの原因は誤りである。校了時点の記述はそのまま残すが、以下が事実である。

- **「Google Places APIへの接続がセッション制限でブロック」は researcher の推測であり、事実ではない。** Places API にはリクエストが一度も届いていない。Google側の障害でもレート制限でもセッション制限でもない
- 真因は工程2のツール許可を `Bash` からパターン照合(`Bash(curl *)` 等)へ絞ったことである。Places はキーの受け渡しに `-H "X-Goog-Api-Key: $GOOGLE_PLACES_API_KEY"` が必要だが、シェル変数展開を含むコマンドは静的検証できずプレフィックス照合が拒否する(CIログの拒否理由は `Contains simple_expansion`)。CIには承認する人間がいないためそのまま落ちた。同じ工程でキー不要の NDL/openBD への curl は十数回すべて成功している
- **したがって「Places APIは障害を起こしやすい」という前提は成り立たない。上の項の末尾にある「発注前の疎通確認」の検討事項は取り下げる。** vol.004・vol.005 では Places から 8件・16件の候補を取得できており、この構成では毎回100%失敗する種類の不具合だった
- 対応済み(PR #21): Places 検索は `scripts/places_search.py` 経由に変更した。キーはスクリプトが環境変数から直接読み、コマンドラインには載らない。researcher が curl で Places を叩くこと、キーの存在を自分で確認することは禁止した
- **教訓: researcher は `ng` の理由に推測した原因を書かない。** ツールが返したエラーメッセージをそのまま転記する。vol.006 では「セッション制限」という実在しない原因が誌面と本申し送りに載り、一日分の誤った前提を残した。`places_search.py` は API 障害(`error` 非null)と候補ゼロ(`error` がnullで `places` 空)を出力で区別するので、以後はこれを転記すれば足りる
